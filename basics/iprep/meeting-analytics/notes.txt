3 different design approaches
--------------------------

Take A.  Rule-based generation

See  basics\iprep\meeting-analytics\generate_rule_based_taxonomy.py

Pure rule-based approach to inferring call types


1. load_records - read json files and load into python MeetingRecord objects

2. create_schema - create postgres SCHEMA
 Create the Postgres schema and analytical tables.

  The tables are designed around the assignment's questions rather than the
  original JSON file boundaries:

  see basics\iprep\meeting-analytics\use_cmds.sql

3.  load_functional_tables
  Insert meeting records and derived taxonomy/sentiment rows into Postgres.

  For each MeetingRecord, this function:

  1. Inserts meeting metadata and participants.
  2. Inserts summary, normalized topics, action items, and key moments.
  3. Inserts transcript lines with sentence-level sentiment.
  4. Runs infer_themes() to populate meeting_themes.
  5. Runs infer_call_type() to populate call_types.
  6. Aggregates transcript sentiment and key-moment counts into sentiment_features.

  This is the point where taxonomy generation becomes concrete database
  evidence. Every theme and call-type row is derived from explicit topics,
  key-moment types, summary text, and keyword rules.

Implementation: basics\iprep\meeting-analytics\take_c_semantic_clustering.py

FAQ: basics\iprep\meeting-analytics\take_c_faq.txt

Pipeline: Extract topics -> Dedup (exact + fuzzy) -> Embed (nomic-embed-text)
          -> UMAP (10-dim) -> HDBSCAN -> LLM label clusters -> assign meetings
          -> infer call types -> write outputs to cluster_work_c/

Dry-run verified: 100 meetings, 600 raw topics -> 351 exact-dedup -> 343 fuzzy-dedup
